Decisions

# Decision Alternatives Considered Rationale

1 Enhance existing Appointment entity Create new Agendamento entity Reuses calendar integration, avoids duplication, Delta 7 decision
2 Use linkParent field with entityList Put entityList on belongsToParent link Follows EspoCRM pattern from BpmnProcess/WorkflowLogRecord - entityList goes on field, not link
3 Reuse Global CascadeDelete hook Create module-specific cascade hooks Avoids code duplication; Global CascadeDelete reads from metadata config
4 Profissional.userId optional Required userId Supports external specialists without system access, Delta 12
5 No ProcedimentoBase entity files Create abstract entity EspoCRM doesn't support entity inheritance; ProcedimentoBase is a naming convention only, Delta 2
6 Sessao.dosagemAplicada is authoritative ProcedimentoRealizado stores dosage Professional fills dosage during session, ProcedimentoRealizado gets snapshot, Delta 4
7 Fase 0 = minimal vertical slice Implement all entities at once Validate end-to-end flow before expanding scope
8 Stream enabled on operational entities No stream Enables audit trail and notes on key entities, Delta 11
9 Appointment status uses English enum values Portuguese enum values Avoids data migration for existing Appointment records; Portuguese display via i18n overlay
10 location field becomes optional on Appointment Keep required Avoids dual-required-field UX conflict with new unidadeId (Unidade); location is deprecated in favor of unidadeId
11 All entities with team-based ACL include teams field Rely on auto-creation EspoCRM team-based ACL requires explicit teams linkMultiple field in entityDefs
12 All entity scopes include hasTeams: true Rely on implicit detection EspoCRM OnlyTeam select filter requires hasTeams: true in scope metadata for team-based ACL filtering
13 Child entities cascade teams from parent via beforeSave hooks (order=1) Rely on user default team Follows Chatwoot CascadeTeamsFromAccount pattern; ensures programmatically-created children inherit correct tenant isolation
14 SeedRole.php updated with FeatureClinica entity permissions No role seeding Tenant and tenant-admin roles must define permissions for all new entities; follows established pattern from Chatwoot/FeatureCredential modules
15 ProcedimentoConsulta.unidadeId is optional (not required) Keep unidadeId required Allows tenant-wide procedure catalogs (unidadeId=null, team-scoped only) or unit-specific ones; adapts to each tenant reality
16 Replace tenant-variable enums with entity-backed team-scoped lookups Keep as fixed enums EspoCRM Entity Manager changes enums globally, not per-tenant; entity-backed lookups (TipoProfissional, Especialidade, CanalAgendamento, UnidadeDosagem) allow each tenant to manage their own catalog. Status enums remain fixed because hooks depend on specific values
17 Use Contact directly as patient entity Create Paciente entity linked to Contact with foreign fields Paciente entity provides a clinical-domain boundary; foreign fields (name, emailAddress, phoneNumber) display Contact data without duplication; patient-specific fields (cpf, dataNascimento, sexo) live on Paciente, not Contact

## Standard Field Contract

All new entities created by FeatureClinica include the following standard fields (not repeated in each entity section below):

- `createdAt` (datetime, readOnly)
- `modifiedAt` (datetime, readOnly)
- `createdBy` (link to User, readOnly)
- `modifiedBy` (link to User, readOnly)
- `teams` (linkMultiple to Team, required for team-based ACL, layoutRelationshipsDisabled: true)

And corresponding standard links:

- `createdBy` (belongsTo User)
- `modifiedBy` (belongsTo User)
- `teams` (hasMany Team, relationName: "entityTeam", layoutRelationshipsDisabled: true)

#### 9. Paciente Entity (STANDARD COMPLEXITY - linked to Contact with foreign fields, Decision 17)

**Resources/metadata/entityDefs/Paciente.json** - Patient entity linked to Contact:

- contactId (link to Contact, required)
- name (varchar, maxLength: 255, readOnly: true) -- stored, synced from Contact.name via SyncNameFromContact hook. Required for pacienteName cached columns in Jornada/Atendimento/Appointment.
- Foreign fields from Contact (using EspoCRM `link`/`field` convention per `metadata/fields/foreign.json`):
  - contactName (type: foreign, link: contact, field: name, readOnly: true)
  - emailAddress (type: foreign, link: contact, field: emailAddress)
  - phoneNumber (type: foreign, link: contact, field: phoneNumber)
- Patient-specific fields:
  - cpf (varchar, unique index)
  - dataNascimento (date)
  - sexo (enum: M | F | Outro)
- Links: contact (belongsTo Contact), jornadas (hasMany Jornada, foreign: paciente), atendimentos (hasMany Atendimento, foreign: paciente), appointments (hasMany Appointment, foreign: paciente)
- Indexes: unique on cpf, index on contactId
- collection: orderBy: createdAt, order: desc, textFilterFields: ["name", "cpf", "contact"]

Paciente uses a stored `name` varchar field (readOnly) synced from Contact.name via `SyncNameFromContact` hook (order: 1, beforeSave). The data model (stored `name` + foreign `contactName`) is inspired by ChatwootContact's entityDefs structure, but the sync mechanism is NEW: ChatwootContact gets its name from the Chatwoot API sync process, whereas Paciente actively copies from the linked Contact entity via a beforeSave hook. The stored field is required because EspoCRM caches link display names (e.g., `pacienteName` column in Jornada/Atendimento) from stored varchar fields, not from foreign-typed fields. Note: the canonical entityDefs convention for foreign fields is `link`/`field` (per `metadata/fields/foreign.json`), not `relation`/`foreign`.

> **IMPLEMENTATION WARNING:** ChatwootContact.json uses legacy `relation`/`foreign` params for foreign fields. Do NOT copy that syntax. Use Espo core convention: `link` and `field` params (per `metadata/fields/foreign.json`). Example: `{"type": "foreign", "link": "contact", "field": "name"}`.

**Resources/metadata/scopes/Paciente.json** - entity: true, tab: true, stream: true, hasTeams: true, module: "FeatureClinica", type: "Base"
**Resources/metadata/clientDefs/Paciente.json** - Client config with relationshipPanels for jornadas, atendimentos, appointments
**Resources/metadata/aclDefs/Paciente.json** - ACL: create: "team", read: "team", edit: "team", delete: "admin"
**Resources/layouts/Paciente/detail.json** - Detail view with contact link, foreign fields, patient-specific fields
**Resources/layouts/Paciente/list.json** - List view with name (foreign), cpf, dataNascimento, contact
**Resources/i18n/pt_BR/Paciente.json** - Portuguese labels: Paciente/Pacientes, field labels (cpf, dataNascimento, sexo, contato, emailAddress, phoneNumber)
**Resources/i18n/en_US/Paciente.json** - English labels
**Hooks/Paciente/SyncNameFromContact.php** - Order: 1. On beforeSave, if contactId is set or changed, loads Contact entity and copies Contact.name to Paciente.name. New pattern: beforeSave hook syncs stored name from linked Contact. Data model inspired by ChatwootContact (stored name + foreign contactName) but sync mechanism is new. Ensures pacienteName cached columns in Jornada/Atendimento/Appointment are populated correctly. If `contactId` is set but the Contact entity cannot be loaded (deleted or inaccessible), the hook should log a warning via `$GLOBALS['log']->warning()` and leave `name` unchanged (preserving the last synced value). This prevents a hard error on save while maintaining data integrity.
**Hooks/Paciente/ValidateUniqueCpf.php** - Order: 9 (default). On beforeSave, if `cpf` is set or changed, queries for existing Paciente with same `cpf` (excluding current entity by id). If found, throws `Espo\Core\Exceptions\Conflict` with user-friendly message. This provides a clean validation error before the database unique constraint fires.

### Multi-Tenant & Customization Strategy

**Team-based ACL (Teams as Tenants):** Every entity includes `teams` linkMultiple and `hasTeams: true` in scope metadata. EspoCRM's `OnlyTeam` select filter enforces data isolation automatically. Each Team represents a tenant ("Permissão / Acesso").

**CascadeTeams pattern:** Child entities created programmatically inherit teams from their parent via beforeSave hooks (order=1), following the Chatwoot `CascadeTeamsFromAccount` pattern:
- Sessao inherits from Jornada (`CascadeTeamsFromJornada`)
- ProcedimentoRealizado inherits from Atendimento (`CascadeTeamsFromAtendimento`)
- Atendimento auto-created from Appointment inherits teams via `CreateAtendimentoOnRealizado`

**Entity-backed lookups (per-tenant flexibility):** Tenant-variable classifications use team-scoped lookup entities instead of fixed enums. Each tenant manages their own catalog as records:
- `TipoProfissional` -- replaces Profissional.tipo enum. Includes `requerCRM` boolean for CRM validation logic.
- `Especialidade` -- replaces Profissional.especialidades array. Linked via many-to-many.
- `CanalAgendamento` -- replaces Appointment.canalAgendamento enum. Each tenant defines their scheduling channels.
- `UnidadeDosagem` -- replaces unidadeDosagem enum on Sessao/ProcedimentoRealizado. Each tenant defines their dosage measurement units.

**Status enums remain fixed:** `Sessao.status`, `Jornada.status`, `Atendimento.status`, `Appointment.status` stay as enums because hooks contain business logic tied to specific values (e.g., "Realized" triggers Atendimento creation).

**Remaining enums (universal, not tenant-variable):** `Profissional.crmUf` (Brazilian states) and `Paciente.sexo` (M/F/Outro) are standard classifications that do not vary by tenant.

**ProcedimentoConsulta scope:** When `unidadeId` is null, the procedure is available tenant-wide (team ACL only). When set, it is restricted to a specific Unidade. This allows each tenant to organize their procedure catalog differently.

**SeedRole integration:** All FeatureClinica entities must be registered in `SeedRole.php` with appropriate permissions for `tenant`, `tenant-admin`, and `tenant-user-api` roles. Paciente is currently missing from SeedRole.php and must be added (see Files to EDIT section). Paciente has create/read/edit/stream for tenant users and full CRUD+stream for tenant-admins. Lookup entities (TipoProfissional, Especialidade, CanalAgendamento, UnidadeDosagem) are read-only for tenant users and full CRUD for tenant-admins.

### Risk Mitigation Notes

1. **linkParent Pattern:** Ensure entityList is on the field definition, NOT on the belongsToParent link. The v2 audit caught this error. For hasChildren links on parent entities, use `"foreign": "procedimento"` (the belongsToParent link name on the child). Do NOT use `foreignKey`/`foreignType` -- these properties do not exist in EspoCRM's hasChildren type. Confirmed pattern: `{"type": "hasChildren", "entity": "...", "foreign": "procedimento"}`.
2. **Appointment Status Migration:** `Rebuild/MigrateAppointmentStatus.php` is in the mandatory file manifest. Registered in rebuild.json so it runs automatically on `php rebuild.php`. Idempotent -- safe to re-run.
3. **SyncProfissionalToAssignedUsers:** This new hook (order 5) runs BEFORE the existing SyncAssignedUsers hook (default order 9 per HookManager::DEFAULT_ORDER). Order 5 chosen because: CascadeTeams hooks use order 1, validation hooks use order 9, and 5 sits cleanly between them for pre-processing bridges. It reads profissionalId, loads the Profissional entity, and injects the userId into professionalsIds. SyncAssignedUsers then picks it up for calendar display without any changes to the Global module.
4. **Composite Indexes:** All linkParent fields require composite indexes: `{"procedimento": {"columns": ["procedimentoType", "procedimentoId"]}}`. All FK fields get individual indexes.
5. **CascadeDelete Configuration:** Add cascadeDelete config to entityDefs, NOT a separate hook. Example: `"cascadeDelete": {"links": ["sessoes", "procedimentosRealizados"]}`
6. **Profissional.userId Optional:** The `required: false` on userId field allows external specialists. Hook validates userId exists if provided. When userId is null, the professional won't appear on calendar (no bridge possible).
7. **location Deprecation:** The overlay sets `location` to `required: false`. Existing appointments continue to work. New appointments use `unidadeId` (Unidade entity) instead. Full migration of location->unidadeId is deferred to "Fase Future".
8. **teams Field & Multi-Tenant:** All new entities include `teams` linkMultiple and `hasTeams: true` in scope for team-based ACL. On entity creation, teams auto-populate from the creating user's default team. Child entities (Sessao, ProcedimentoRealizado) cascade teams from parent via beforeSave hooks (order=1). Auto-created Atendimento inherits teams from Appointment.
9. **Standard Audit Fields:** All new entities include createdAt, modifiedAt, createdBy, modifiedBy. These are standard EspoCRM fields and should be included in layouts (typically in side panel or footer).
10. **Convenio Forward Reference:** Jornada.convenioId and Appointment.convenioId are defined as optional links in Fase 0. The Convenio entity doesn't exist yet (Fase 1). Both fields are HIDDEN from layouts (omitted from detail.json/list.json) until Fase 1 to avoid confusing users. The DB columns exist but are unused. This avoids a schema migration when Convenio is introduced. If a Convenio ID is written directly via API before the Convenio entity exists (Fase 1), EspoCRM will store the foreign key but display an empty link (no entity to resolve). This is harmless and self-corrects once Fase 1 is deployed. No server-side validation blocks this edge case since the field is optional and not validated against entity existence by default.
11. **Profissional.userId null behavior:** When Profissional.userId is null, SyncProfissionalToAssignedUsers skips the bridge -- the professional won't appear on calendar views. This is expected behavior for external specialists without system access (Decision 4). No error is thrown; the hook simply does not add a user to professionalsIds.
