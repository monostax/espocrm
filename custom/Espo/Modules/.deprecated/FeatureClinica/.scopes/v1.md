File Manifest: EspoCRM Clinical Management System v3.0
Decisions
| # | Decision | Alternatives Considered | Rationale |
|---|----------|------------------------|-----------|
| 1 | New module Saude for all entities | Multiple modules per layer, extend existing modules | Single module simplifies dependencies, cross-entity relationships, and deployment |
| 2 | Use Contact entity as Paciente | Create new Paciente entity | Contact already has person fields, email, phone, address - avoids duplication |
| 3 | ProcedimentoBase is conceptual only | Real entity with inheritance | EspoCRM doesn't support entity inheritance; linkParent entityList provides the polymorphic behavior |
| 4 | All linkParent references use entityList with all 5 procedure types | Individual entityList per use case | Consistency across all relations, easier to maintain when adding new procedure types |
| 5 | unidadeId on all operational entities | Global scope with filtering | Enables multi-unit reporting, permissions, and stock control natively |
| 6 | Orcamento as separate entity | Status in LancamentoFinanceiro | Preserves commercial pipeline tracking, versioning, and rejected quotes |
| 7 | dosagemAplicada in Sessao and ProcedimentoRealizado | JSON field | Structured data enables dosage curve reports and validation |
| 8 | valorCobrado snapshot in ProcedimentoRealizado | Live reference to TabelaDePrecos | Historical immutability - price changes don't retroactively affect records |
| 9 | Prescricao with PrescricaoItem | JSON array in Prescricao | Enables multiple medications per prescription, each with own dosage/instructions |
